# Философский и архитектурный аудит mk-rust

*Дата: 2026-06-07*
*Задача: соответствует ли mk-rust духу Plan 9 mk и cat-v.org?*

---

## 1. Верность mk (Faithfulness)

### Синтаксис mkfile — практически 1:1

| Фича mk | В mk-rust | Статус |
|---------|-----------|--------|
| `target: prereqs` | То же самое | ✅ |
| `target:V: prereqs` | `target:V: prereqs` | ✅ |
| Метаправила `%.o: %.c` | То же самое | ✅ |
| `&` метаправило | То же самое | ✅ |
| `R:` regex-правила | То же самое (через крейт `regex`) | ✅ |
| `$VAR` / `${VAR}` | То же самое | ✅ |
| `${VAR:A%B=C%D}` (немлисты) | То же самое | ✅ |
| `< file` include | То же самое | ✅ |
| `<| cmd` include | То же самое | ✅ |
| `lib(member)` архив | То же самое | ✅ |
| `P:` атрибут (кастомный comparison) | То же самое | ✅ |
| `clean:V:\n\trm -f *.o` | То же самое | ✅ |

**Отличий в синтаксисе нет.** Парсер Plan 9 mkfile без изменений. Это был ключевой критерий.

### Модель выполнения — 6 фаз из mk

Plan 9 mk: Parse → DAG → Closure → Prune → Verify → Execute

| Фаза | В mk-rust |
|------|-----------|
| **Parse** | `lex::tokenize()` + `parse::parse()` |
| **DAG** | `graph::build_graph()` — строит узлы и рёбра из конкрентных правил |
| **Closure** | Транзитивное замыкание метаправил — в `build_graph()` |
| **Prune** | `graph::prune_vacuous()` — удаляет meta-рёбра если есть concrete |
| **Verify** | `graph::cyclechk()` + `ambiguous()` — циклы, неоднозначности |
| **Execute** | `sched::execute()` — топологический порядок, параллель |

Модель верна. Нюанс: plan9port выполняет closure до построения полного графа, у нас closure происходит в процессе построения (ленивое). Результат тот же.

### Преднамеренно не реализовано

| Фича | Причина |
|------|---------|
| **rc shell** | Никто не использует rc вне Plan 9. Duckscript — замена с лучшей производительностью. |
| **membername** утилита | Внешний rc-скрипт, не часть mk. |
| **$OBJ / $O** (Plan 9 ABI) | Аппаратно-зависимая фича Plan 9. Не имеет смысла вне Plan 9. |
| **Динамический mkfile из stdout рецепта** | Упоминается в Hume87 как идея, никто не использует. |
| **Циклические зависимости** (make-style) | mk их принципиально не поддерживает — и мы тоже. |

Все intentional — ни одна не нужна для работы с реальными mkfile на Unix.

### Что добавили сверх mk

| Фича | Почему |
|------|--------|
| **Duckscript** | Вместо rc. Feature-gated, не увеличивает размер core. Позволяет рецептам без shell. |
| **Крейт `regex`** | Для `R:` метаправил. Plan 9 использует свой regexp(7) — Rust-эквивалент. |
| **Крейт `clap`** | CLI-парсер. Единственная reasonable dependency для CLI. |
| **`$MKFLAGS` / `$MKARGS`** | Полезно для рецептов, не противоречит духу. |

**Feature creep минимален.** Ничего из этого не обязательно для core — всё опционально.

## 2. Дух cat-v.org

> *«Complexity is the bane of all software, simplicity is the most important quality.»*

### Метрики простоты

| Метрика | mk-rust | plan9port mk | GNU Make |
|---------|---------|--------------|----------|
| Язык | Rust | C (Plan 9 dialect) | C |
| LOC | 7,019 | ~4,350 | ~32,000+ |
| Файлов исходников | 14 | 22 (.c) | 100+ |
| Зависимостей (core) | 2 | 0 (Plan 9 libs) | десятки |
| Мануал | 302 строки | 200+ строк | 183 страницы |
| Модулей | 11 | 1 (monolith) | множество |

**mk-rust НЕ проще C-версии по LOC** (7K vs 4K). Но это включает тесты и документацию внутри модулей. Чистого продакшн-кода: ~4,500 LOC — примерно на уровне plan9port C.

**mk-rust ГОРАЗДО проще GNU Make** — на порядок меньше кода, на порядок меньше фич, нет встроенной магии (автоматические правила, `.SUFFIXES:`, `$(call ...)`).

### Признаки over-engineering

**Жёлтые флаги:**
1. **`sched.rs` (1,053 LOC)** — содержит и sequential и parallel execution + тесты. Возможно, стоит разделить на `sched/serial.rs`, `sched/parallel.rs`.
2. **`graph.rs` (1,393 LOC)** — делает слишком много: DAG-строительство, метаправила, closure, pruning, NREP, P-атрибут, архивы, cycle detection. Кандидат на разделение.
3. **`#[allow(clippy::too_many_arguments)]`** — дважды в коде. Симптом: функции принимают слишком много параметров. В идеале → `BuildContext` struct.

**Зелёные флаги (что хорошо):**
1. **Только 2 зависимости в mk-core** — `thiserror` + `regex`. Это исключительно хорошо для современного Rust-проекта.
2. **Duckscript feature-gated** — не увеличивает размер core.
3. **Нет async, нет tokio** — строго синхронный, как и сам mk.
4. **Агенты-субагенты** — архитектура с субагентами для исследования, кодера, ревьювера — это в духе Unix-way (как pi-way).

### Сопротивление feature creep

| Было в плане | Реальность | Вердикт |
|-------------|------------|---------|
| `serde` optional | **Нет** | Убрали — не нужно |
| `crossbeam` | **Нет** | Используем `std::thread::scope` |
| `duct` | **Нет** | Используем `std::process::Command` |
| `glob` | **Нет** | Simple path matching |
| `tempfile` | **Нет** | Не понадобился |
| `env_logger` | **Нет** | Debug через `-d` флаг |

**Реальный граф зависимостей меньше планового.** Это признак здоровой инженерной дисциплины.

## 3. Архитектурная целостность

### Модульная структура

```
lex → parse → graph → sched → shell/recipe
   ↑         ↑         ↑
   │         │         └── var (переменные)
   │         │         └── attr (attributes)
   │         │
   │         └── include (рекурсивные includes)
   │
   └── archive (парсинг lib(member))
```

**Поток данных чистый:** каждый модуль принимает output предыдущего, производит следующий тип данных. Никаких циклических зависимостей.

**Проблема: размытые границы между parse, var и graph**
- `parse::Rule` хранит строки-переменные без раскрытия (F-045: rule headers evaluated at parse time). Хорошо.
- `var::Scope` существует независимо от parse — это правильно.
- Но метаправила в graph требуют, чтобы parse уже отметил `is_metarule`. Это ок — parse делает это в момент разбора.

### Shell trait — удачный дизайн

```rust
pub trait Shell: Send + Sync {
    fn name(&self) -> &str;
    fn execute(&self, recipe: &str, env: &HashMap<String, String>, dir: &Path) -> Result<ShellResult, ShellError>;
}
```

Минималистичный интерфейс, легко mock'ать, легко добавлять новые реализации (ShShell, DuckShell). Send + Sync — позволяет использовать в parallel execution без Arc<Mutex<>>.

**В плане была RcShell — заменили на DuckShell.** Правильное решение: rc умирает, duckscript живёт.

### DAG-модель с индексами (NodeIndex, ArcIndex)

```rust
pub struct NodeIndex(pub usize);
pub struct ArcIndex(pub usize);
pub struct Graph {
    pub nodes: Vec<Node>,
    pub arcs: Vec<Arc>,
}
```

**Это лучшее решение.** Arena allocation без Rc/RefCell — быстрая, cache-friendly, без циклических ссылок. Единственный минус: `NodeIndex` — просто `usize`, нет type safety против разных arena. Но для mk это не проблема — DAG один.

## 4. Чего не хватает vs Что лишнее

### 3 вещи, которые ДОЛЖНЫ БЫТЬ

1. **Проверка mkfile на уровне файла `-v`**
   Сейчас если mkfile синтаксически неверен, ошибка при запуске. `mk -v` (validate) был бы полезен. Некоторые mkfile (mkenv из plan9port) являются dual-format (shell + mk) — валидация помогла бы отлавливать плохие mkfile.

2. **Тест на реальном mkfile (testdata/integration)**
   У нас есть 43 тестовых mkfile в `testdata/external/`, но ни один не запускается автоматически. Нужен скрипт, который прогоняет `mk -n` на каждом и проверяет, что парсинг не падает.

3. **`--color` / авто-цвет**
   plan9port mk поддерживает цветной вывод. Наш — нет. Полезно, не сложно (isatty + ANSI).

### 3 вещи, которые ЕСТЬ НО НЕ НУЖНЫ

1. **Duckscript (пока)**
   Работает, feature-gated, но для mk-rust основная ценность — совместимость с mkfile, где рецепты это shell. Duckscript — нишевое улучшение. Не bloat, потому что опционально, но и не нужно для первой версии.

2. **Archive aggregates `lib(member)` (пока)**
   Правильно работает, но хардкорно plan9-специфично. Никто вне Plan 9 не использует `ar` archives через mk. Для GNU/Linux проектов это избыточно.

3. **P attribute (custom comparison)**
   Мощная фича, но в реальности используется редко. Plan 9 порт имеет её, мы тоже. Но можно было отложить.

## 5. Вердикт

**mk-rust — достойный порт Plan 9 mk на Rust.** Синтаксис mkfile совместим 1:1. Модель выполнения (6 фаз, transitive closure, NPROC параллелизм) верна. Feature creep минимален (опциональный duckscript, 2 зависимости в core). Код простой (7K LOC, 11 модулей, без unsafe, без async).

**Единственное, что стоит исправить перед релизом:** разбить `graph.rs` и `sched.rs` на подмодули, и запустить хотя бы один тест на реальном mkfile из testdata/external (например, ctSkennerton/test1.mk) для сквозной верификации.

*Всё остальное — косметика.*
