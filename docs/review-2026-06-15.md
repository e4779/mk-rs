## 🔴 Bugs

### 1. `$NREP` variable defined but never read — graph.rs / cli

**File:** `crates/mk-core/src/var.rs:362`, `crates/mk-core/src/graph.rs:206-214`, `crates/mk-cli/src/main.rs`

**Severity:** 🔴 Bug (user-visible)

The `$NREP` variable is set as a builtin default:
```rust
s.set("NREP", "1", Precedence::Builtin);
```
But `build_graph()` never reads the variable scope; it hardcodes `nrep=1`:
```rust
pub fn build_graph(stmts: &[Stmt], target_names: &[String]) -> Result<Graph, GraphError> {
    build_graph_with_nrep(stmts, target_names, 1)  // hardcoded, not $NREP
}
```
The CLI (main.rs) also never passes an `nrep` value. A user who sets `NREP=2` in their mkfile will see no effect — the metarule recursion depth never changes.

**Impact:** Metarules that require NREP>1 for transitive closure will silently fail to expand correctly. The spec (F-056) says `$NREP` limits how many times a metarule is used when generating prerequisites, defaulting to 1.

**Suggested fix:** Read `$NREP` from the variable scope in the CLI glue, and pass it to `build_graph_with_nrep`. The `build_graph` public API should either accept `nrep` as a parameter or read it from the scope internally.

### 2. `prune_vacuous` does not prune unreachable subgraphs — graph.rs

**File:** `crates/mk-core/src/graph.rs:637-648`

**Severity:** 🔴 Bug (F-060 incomplete)

The `prune_vacuous` function only removes metarule arcs when a concrete rule exists for the same target:
```rust
fn prune_vacuous(graph: &mut Graph) {
    for node in &mut graph.nodes {
        let has_concrete = node.arcs_in.iter().any(|&ai| !graph.arcs[ai.0].is_meta);
        if has_concrete {
            node.arcs_in.retain(|&ai| !graph.arcs[ai.0].is_meta);
        }
    }
}
```
This does NOT prune nodes that are unreachable from the requested targets (F-060 — "After closure, mk prunes parts of DAG not needed for desired target"). If a metarule generates deep transitive closure chains that are not on any path to the requested target, those nodes remain in the graph.

**Impact:** Builds may have unnecessary nodes in the DAG, though execution may still be correct since only stale nodes in dependency paths execute. The TRACEABILITY.md marks F-060 as `—` (not started), confirming this gap.

**Suggested fix:** After transitive closure, run a reachability pass from the target set and remove all unvisited nodes.

## 🟡 Spec Compliance Gaps

### 3. F-042: Command-line variable assignments (`mk VAR=value`) — Not implemented

**File:** `crates/mk-cli/src/main.rs`

The CLI parses only targets and flags. There is no code to split `VAR=value` from the `targets` vector and inject them into the variable scope at `Precedence::CommandLine`. TRACEABILITY.md marks this as `—`.

**Suggested fix:** Add logic in `main.rs` to separate `VAR=value` entries from target names, call `scope.set(name, value, Precedence::CommandLine)` for each.

---

### 4. F-052: `-s` flag (force sequential) — Not in CLI

**File:** `crates/mk-cli/src/main.rs`

No `-s` / `--sequential` CLI flag. The scheduler reads `$NPROC` from the environment scope, but there's no way to override parallelism from the command line. The `-s` flag should force `nproc=1` regardless of `$NPROC`.

**Suggested fix:** Add `-s` flag to Cli struct; when set, force `nproc=1` in SchedOptions.

---

### 5. F-048: `-w` flag (what-if) — Stub only

**File:** `crates/mk-cli/src/main.rs:99`

The `-w` flag is parsed but assigned to `cli.whatif` which is only read by `let _ = &cli.whatif;` (line 109). No what-if behavior is implemented in graph.rs staleness checker or the scheduer. TRACEABILITY.md marks this as `—`.

**Suggested fix:** Either implement what-if mode in the staleness checker (pretend listed files have current time), or remove the stub flag until ready.

---

### 6. F-067: Multiple `-f` flags (mkfile concatenation) — Not supported

**File:** `crates/mk-cli/src/main.rs:47`

The `-f` flag is a single `PathBuf`:
```rust
#[arg(short = 'f', default_value = "mkfile")]
file: PathBuf,
```
The spec says "Multiple `-f` flags allowed (concatenated)". With clap's single-value binding, only the last `-f` wins.

**Suggested fix:** Change to `Vec<PathBuf>` and concatenate all specified mkfiles.

---

### 7. F-050: `-d` debug flag — Prints stubs but enables nothing

**File:** `crates/mk-cli/src/main.rs:97`

The `-d` flag is parsed and prints messages like `mk: debug: parsing enabled` but doesn't actually enable debug-level logging anywhere in the pipeline. TRACEABILITY.md marks it as `✓` (Phase 3), but the implementation is a stub.

**Suggested fix:** Wire debug flags to set verbosity in lexer/parser/graph/scheduler, or update TRACEABILITY to show `◐`.

## 🟡 Missing Test Coverage

### 8. Recipe-time builtin variables: `$nproc`, `$MKFLAGS`, `$MKARGS`

**File:** `crates/mk-core/src/recipe.rs:140-156`

The recipe `run()` function injects `target`, `prereq`, `newprereq`, `pid`, `alltarget`, `newmember`, and `stem` into the recipe environment. But `$nproc` (process slot number), `$MKFLAGS` (mk option arguments), and `$MKARGS` (target arguments) are **not injected** even though they're defined in the spec (F-037, F-054, F-055) and TRACEABILITY says ✓ for all three.

**Impact:** Shell recipes in parallel builds can't use `$nproc` for temp file naming.

**Suggested fix:** Add `$nproc` to recipe environment in `run()`. Wire `$MKFLAGS` and `$MKARGS` from SchedOptions.

---

### 9. No tests for `E` attribute behavior (continue on error)

**File:** `crates/mk-core/src/attr.rs` (E defined), `crates/mk-core/src/sched.rs` (keep_going tested)

The `E` attribute (`ATTR_EXCLUSIVE`) is defined but the scheduler uses `opts.keep_going` (the `-k` flag) rather than checking individual rule attributes. The `E` attribute should cause mk to continue even if the recipe for this specific target fails, regardless of `-k`.

TRACEABILITY marks this as `◐` (attr parsing done, behavior pending).

**Suggested fix:** Add test verifying that a target with `:E:` attribute fails but doesn't halt sibling targets.

---

### 10. No tests for `U` attribute (unconditionally updated)

The `U` attribute is parsed but its semantic — "target is considered updated even if recipe didn't change it" — has no test verifying that after a recipe runs on a `U` target, it's not re-run even if the file's mtime didn't change.

**Suggested fix:** Add test: target with :U: attribute, recipe that doesn't create the file. After running once, staleness check should say "not stale".

---

### 11. Variable scope for includes — no child-scope tests

**File:** `crates/mk-core/src/include.rs`

The include system parses included mkfiles but there's no test for variable isolation: included mkfile assignments should go into a child scope that shadows but doesn't leak to the parent. The `Scope::with_parent` API exists but isn't wired through `include_file`.

**Suggested fix:** Add test: parent has `FOO=parent`, included file sets `FOO=child`, after include verify `FOO` is still `parent`.

---

### 12. Multiple targets behavior ($target vs $alltarget)

There's no test for a rule with multiple targets:
```mkfile
a b: c d
	cp $prereq $target
```
When only `a` is stale but `b` is up-to-date, `$target` should contain only `a` and `$alltarget` should contain `a b`. Currently, `build_recipe` always sets `all_targets: vec![node.name.clone()]` — a single-element list — regardless of multi-target rules.

**Impact:** `$alltarget` in recipes is incorrect for multi-target rules.

### 17. `MkError` / `MkResult` are dead code

**File:** `crates/mk-core/src/error.rs:155`

```rust
pub type MkResult<T> = Result<T, MkError>;
```
The `MkError` enum wraps all sub-error types, but every module in `mk-core` returns its own specific error type (`ParseError`, `GraphError`, etc.) directly. The unified error type exists for public API but is never used — modules use their own types. Even the CLI uses `Box<dyn std::error::Error>`.

**Suggestion:** Either use `MkError` everywhere (consistent), or remove it (YAGNI). The current setup wastes the `thiserror` derive.

## 🟢 Performance / Architecture

### 18. `expand_globs` allocates even for no-glob patterns

**File:** `crates/mk-core/src/graph.rs:149-167`

Every prerequisite is checked for glob characters and cloned into a new `Vec<String>` even if none need expansion. The allocation cost is proportional to the number of prereqs which is typically small, but unnecessary.

**Suggested fix:** Return `Cow<[String]>` or use `smallvec`, or return early when no prereqs contain glob chars.

---

### 19. Metarule matching on every build node does redundant work

**File:** `crates/mk-core/src/graph.rs:378-436`

The `build_node` function iterates all metarules and regex rules for every unresolved target. If there are M metarules and N nodes, this is O(M×N). With large mkfiles this can be slow. Caching metarule-indexed patterns (e.g., pre-compiled regexes, pre-split pattern prefix/suffix) would help.

**Suggestion:** Pre-compile regex rules and pre-split `%`/`&` patterns at graph construction time once, rather than per-node.
